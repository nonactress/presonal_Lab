- 도메인: 인지심리학 → UX/서비스 사용 중 사용자의 멘탈 모델 형성과 이탈 패턴에 적용 가능

# CHUNK 1 — 핵심 개념: 인지 부하(Cognitive Load)란 무엇인가
**검색 키워드 힌트:** 인지 부하, cognitive load, 사용자 멘탈 모델, UI 복잡도

인간의 인지 처리 용량(cognitive processing capacity)은 고정되어 있고 제한적이다.
특정 과제가 이 용량을 많이 차지할수록, 다른 처리(학습, 패턴 인식 등)에 쓸 수 있는 용량이 줄어든다.

인지 부하가 높은 상태에서는:
- 문제를 "풀어내는" 것 자체에 급급해 전체 구조를 파악하지 못함
- 스키마(schema) 형성, 즉 "이 상황에서 이렇게 하면 된다"는 패턴 학습이 일어나지 않음
- 결국 같은 과제를 반복해도 숙련도가 오르지 않음

→ **페르소나 적용:** 처음 서비스를 쓰는 사용자는 UI 탐색 자체에 인지 용량을 대부분 소모한다.
이 상태에서 복잡한 정보 입력이나 선택지가 주어지면 "시스템이 충돌"하듯 이탈하거나 포기한다.

# CHUNK 2 — 전문가 vs 초보자의 근본적 차이: 스키마(Schema) 유무
**검색 키워드 힌트:** 초보 사용자, 전문가 사용자, 패턴 인식, 스키마

전문가와 초보자의 차이는 지능이나 노력이 아니라 **스키마의 유무**다.

- **전문가:** 상황을 보는 순간 "이건 이런 케이스"라고 분류하고 바로 행동한다.
  인지 부하가 낮고, 남은 용량으로 더 깊은 처리가 가능하다.
- **초보자:** 상황을 분류할 스키마가 없어서 매번 처음부터 모든 가능성을 탐색한다(means-ends analysis).
  인지 부하가 높고, 탐색 자체에 용량이 소진된다.

스키마는 "이 상태에서는 이 행동이 맞다"는 패턴의 묶음이다.
스키마가 없으면 선택지가 적어도 과부하가 걸린다.

→ **페르소나 적용:**
- 신규 사용자는 스키마가 없기 때문에, 옵션이 3개만 있어도 어디서부터 시작해야 할지 모른다.
- 재방문 사용자는 스키마가 형성되어 있어 같은 화면에서 훨씬 빠르고 자신 있게 행동한다.
- **같은 UI라도 초보와 숙련 사용자가 완전히 다른 지점에서 막힌다.**

# CHUNK 3 — 왜 사용자는 "어렵지 않아 보이는" 기능에서도 막히는가
**검색 키워드 힌트:** UX 이탈, 온보딩 실패, 사용자 막힘 지점, 인지 과부하

Means-ends analysis(수단-목표 분석)는 초보자가 쓰는 기본 탐색 전략이다.
이 전략은 "현재 상태와 목표 사이의 차이를 줄이는 방법"을 매 순간 계산한다.

이 과정에서 사용자는 동시에 다음을 머릿속에 유지해야 한다:
1. 지금 내가 어디 있는가 (현재 상태)
2. 내가 원하는 것이 무엇인가 (목표)
3. 현재 상태와 목표의 차이
4. 사용 가능한 행동들 (버튼, 메뉴 등)
5. 이전에 했던 행동들 (되돌아가지 않기 위해)

이 다섯 가지를 동시에 처리하는 것이 인지 부하의 핵심이다.
항목 하나라도 불분명하면(예: 목표가 뭔지 모르겠음, 버튼이 뭘 하는지 모르겠음) 시스템이 과부하 상태가 된다.

→ **페르소나 적용:**
- "다음에 뭘 눌러야 하는지 모르겠어서" 이탈하는 사용자 → 현재 상태 또는 목표가 불명확한 케이스
- "해봤는데 원하는 결과가 안 나와서" 포기하는 사용자 → 행동과 결과의 연결이 끊어진 케이스
- 두 경우 모두 인지 부하 과잉이 원인이며, 기능 자체의 어려움과는 별개다.

# CHUNK 4 — 인지 부하를 줄이는 설계 원칙: Goal-free / Worked Example 효과
**검색 키워드 힌트:** 온보딩 설계, 튜토리얼, 단계별 가이드, UX 설계 원칙

인지 부하를 낮추는 두 가지 핵심 접근법:

**1. Nonspecific goal (목표를 열어두기)**
"이것을 달성하라"는 명확한 목표 대신, "할 수 있는 것을 탐색해보라"는 방식.
초보자가 목표와 현재 상태의 차이를 계산하는 데 용량을 쓰지 않아도 되므로 인지 부하가 낮다.
탐색 자체에 집중할 수 있어 스키마 형성이 빨라진다.

**2. Worked example (완성된 예시 먼저 보여주기)**
단계별 풀이 과정을 먼저 보여주면, 사용자가 직접 탐색하는 것보다 스키마 형성이 빠르다.
"무엇을 해야 하는지" 고민하는 인지 부하가 없어지므로 "어떻게 작동하는지"에 집중 가능.

→ **페르소나 적용:**
- 초보 사용자 페르소나는 "해보세요" 방식보다 "이렇게 하는 거예요" 예시형 온보딩에서 성공률이 높다.
- 첫 진입 시 "무엇이든 자유롭게"는 실제로는 가장 높은 인지 부하를 주는 방식이다.
- **기능이 단순해도 처음 마주치는 사용자에게는 항상 worked example이 필요하다.**

# CHUNK 5 — 스키마 형성을 방해하는 조건들 (UX 실패 패턴)
**검색 키워드 힌트:** 학습 방해, 반복 사용에도 실력이 안 느는 사용자, UI 복잡도

목표 달성(task completion)과 스키마 형성(학습)은 **서로 다른, 때로는 충돌하는 과정**이다.

사용자가 어떻게든 과제를 완료했다고 해서 다음번에도 잘 할 수 있는 건 아니다.
높은 인지 부하 상태에서 완료한 과제는 스키마를 남기지 않는다.

스키마 형성을 방해하는 조건:
- **과제 완료 압박:** "일단 제출해야 해서" → 구조 파악 없이 버튼만 누름
- **너무 많은 선택지:** 목표와 현재 상태 차이 계산이 복잡해져 용량 초과
- **피드백 부재:** 행동과 결과가 연결되지 않아 패턴 형성 불가
- **오류 복구 어려움:** 잘못 눌렀을 때 원래 상태로 돌아오기 어려우면 탐색 자체를 포기

→ **페르소나 적용:**
- "몇 번 써봤는데도 여전히 어렵다"는 사용자 → 스키마 형성이 안 되고 있는 상태
- 이탈한 사용자가 재방문해도 같은 지점에서 막힌다면 → 해당 지점이 인지 부하 병목

# CHUNK 6 — 전문가/초보 사용자가 같은 UI를 다르게 경험하는 이유
**검색 키워드 힌트:** 숙련 사용자, 초보 사용자, 사용자 세그먼트, UI 인식 차이

전문가(숙련 사용자)는 표면적 특징이 아닌 **구조적 패턴**으로 상황을 분류한다.
초보자는 표면적 특징(버튼 색깔, 위치, 레이블 텍스트)에 의존한다.

예: 같은 "설정" 화면에서
- 숙련 사용자 → "이건 권한 관련 설정이니까 여기 있겠다"고 스키마로 바로 이동
- 초보 사용자 → 모든 메뉴를 순서대로 읽으며 찾음, 레이블이 낯선 용어면 즉시 이탈

이 차이는 동기나 노력의 문제가 아니라 스키마의 유무에서 온다.

→ **페르소나 적용:**
- 숙련 사용자 페르소나: "기능이 어디 있는지 찾는 데 막히지 않음, 기능 자체의 깊이에 불만 가능성"
- 초보 사용자 페르소나: "레이블/아이콘이 낯설거나 다음 단계가 불명확하면 즉시 이탈"
- 두 페르소나가 같은 기능에서 막히는 이유와 지점이 완전히 다르므로 별도로 설계해야 함
