# [문서 메타데이터]
- 출처: Nah (2004), "A study on tolerable waiting time: how long are Web users willing to wait?", Behaviour & Information Technology
- 용도: 사용자 페르소나 생성 시 참고 — 사용자가 응답 지연 앞에서 언제 이탈하는지, 피드백이 이탈 시점에 어떤 영향을 미치는지 예측에 활용
- 도메인: HCI / 웹 UX → 로딩 시간, 응답 속도, 피드백 설계가 사용자 이탈에 미치는 영향

# CHUNK 1 — 핵심 수치: 사용자는 몇 초까지 기다리는가
**검색 키워드 힌트:** 로딩 시간, 이탈 시점, 응답 속도, 대기 한계

이 연구의 핵심 결론: **피드백이 없을 때 사용자의 허용 대기 시간(TWT)은 약 2초**다.

주요 수치 정리:
- **0.1초:** 사용자가 즉각 반응으로 인식하는 한계. 별도 피드백 불필요.
- **1초:** 사용자의 사고 흐름이 끊기지 않는 한계.
- **2초:** 단순 정보 검색에서 대다수 사용자가 이탈을 결정하는 시점.
  61%의 사용자가 피드백 없는 환경에서 2~4초 사이에 포기.
- **15초:** 피드백이 전혀 없을 때의 절대적 상한선.
  이 연구에서 피드백 없는 그룹은 15초 이상 기다린 사용자가 없었음.

→ **페르소나 적용:**
- 로딩/처리 시간이 2초를 넘기는 플로우가 있다면 → 해당 지점이 이탈 병목
- 특히 검색, 조회, 결과 확인 등 **정보 검색형 과제**에서 2초 기준이 엄격하게 적용됨

# CHUNK 2 — 피드백(로딩 인디케이터)의 효과
**검색 키워드 힌트:** 로딩 스피너, 진행 표시, 피드백 설계, 대기 시간 연장

피드백(로딩 바, 스피너 등)은 사용자의 허용 대기 시간을 **극적으로 늘린다.**

실험 결과:
- 피드백 없음 → 첫 번째 로딩 실패 시 평균 대기 13초, 이후 급격히 감소
- 피드백 있음 → 첫 번째 로딩 실패 시 평균 대기 37.6초 (약 3배)

피드백이 효과를 내는 두 가지 이유:
1. **주의 분산(Resource-allocation):** 피드백을 보는 동안 시간이 흐르는 것에 덜 집중하게 됨.
   채워진 시간은 비어있는 시간보다 더 짧게 느껴짐.
2. **불확실성 감소(Uncertainty reduction):** "시스템이 작동 중이다"는 신호가
   "내가 뭔가 잘못한 건 아닐까"라는 불안을 줄여줌.

→ **페르소나 적용:**
- 로딩 인디케이터가 없는 상태에서 처리 시간이 2초 이상이면 → 사용자는 오류로 인식하고 이탈
- 단순한 움직이는 스피너만으로도 허용 대기 시간이 유의미하게 늘어남
- "시스템이 작동 중"이라는 신호 자체가 인지 부하와 불안을 낮추는 UX 장치

# CHUNK 3 — 반복 경험에 따른 허용 대기 시간의 감소
**검색 키워드 힌트:** 반복 이탈, 기대치 하락, 재방문 사용자, 누적 실망

한 번 로딩 실패를 경험한 사용자는 **다음 로딩에서 훨씬 빨리 이탈한다.**

실험 결과 (피드백 없는 그룹):
- 1번째 실패: 평균 대기 13초
- 2번째 실패: 평균 대기 4초 (→ 69% 감소)
- 3번째 실패: 평균 대기 3.3초

이유: 한 번 실패를 경험하면 사용자의 기대치가 낮아지고,
"또 안 되겠지"라는 예상이 빠른 이탈로 이어진다.

→ **페르소나 적용:**
- 플로우 중 한 번이라도 응답 지연을 경험한 사용자는 이후 단계에서 인내심이 급감
- 온보딩 중 첫 번째 느린 응답이 이후 전체 경험의 기대치를 낮춤
- "첫 경험에서의 속도"가 서비스 전체 신뢰도를 결정하는 핵심 변수

# CHUNK 4 — 과제 유형에 따른 허용 대기 시간의 차이
**검색 키워드 힌트:** 과제 유형, 검색 vs 구매, 목적성, 사용자 동기

허용 대기 시간은 사용자가 무엇을 하려는지에 따라 달라진다:

- **목적 없는 탐색(Netsurfing/Browsing):** 느슨한 목표 → 대기 인내심 낮음, 빠르게 이탈
- **정보 검색(Querying):** 명확한 목표 → 대기 인내심 다소 높음
- **구매/결제:** 완료 동기가 강함 → 상대적으로 더 기다릴 의향

단, Selvidge(2003)의 연구는 과제 유형이 허용 대기 시간에 영향을 미치지 않는다는 상반된 결과도 있어 아직 논쟁 중.

→ **페르소나 적용:**
- 목적 없이 탐색 중인 사용자(예: 피드 스크롤, 카테고리 브라우징) → 2초 기준 엄격 적용
- 특정 목적을 가진 사용자(예: 특정 상품 검색, 정보 확인) → 약간의 여유가 있지만 여전히 짧음
- 결제/제출 직전 단계에서의 지연 → 완료 직전 이탈은 가장 치명적인 UX 실패

# CHUNK 5 — 지각된 대기 시간 vs 실제 대기 시간
**검색 키워드 힌트:** 체감 속도, 시간 인식, UX 속도감, 피드백 디자인

사용자는 실제 경과 시간이 아닌 **체감 시간**을 기반으로 이탈 여부를 결정한다.

핵심 발견:
- 사람은 수동적 대기(waiting) 시간을 **실제보다 길게 느끼는** 경향이 있음
- 능동적 활동 시간은 실제보다 짧게 느낌
- 대기 중 무언가를 보여주면(채워진 시간) 체감 시간이 줄어듦

실용적 함의: 실제 로딩 속도를 개선하는 것과 **체감 속도를 개선하는 것은 별개의 전략**이다.
체감 속도 개선 방법:
- 스켈레톤 UI (레이아웃을 먼저 보여주기)
- 점진적 로딩 (일부 컨텐츠를 먼저 표시)
- 로딩 애니메이션 (단순히 "작동 중"임을 알림)
- 기대 대기 시간 명시 ("약 3초 후 완료됩니다")

→ **페르소나 적용:**
- 실제 응답 속도 개선이 어렵다면 → 체감 속도 개선만으로도 이탈률 감소 가능
- 초보 사용자일수록 피드백 없는 대기에서 "내가 뭔가 잘못한 건 아닐까"라는 불안이 더 커짐
- 리터러시 낮은 사용자 페르소나 → 로딩 중 불안으로 인한 중복 클릭 → 오히려 상황 악화
